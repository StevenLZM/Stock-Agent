# 定时股票信息推送 Agent PRD

版本：v0.1  
日期：2026-06-24  
状态：Draft  

## 1. 背景

股票关注者通常需要在固定时间或关键事件发生时快速了解自选股状态，但原始行情、公告、新闻和财务数据分散在不同平台。用户真正需要的不是更多信息流，而是经过筛选、可追溯、低噪音的股票信息摘要。

本产品旨在构建一个定时推送股票信息的 Agent。它按用户配置的股票池、时间表和关注规则，自动采集行情与资讯，识别值得关注的变化，并生成结构化摘要推送到指定渠道。

## 2. 产品定位

本产品是股票信息监控与研究摘要工具，不是自动荐股、自动交易或投资顾问系统。

产品默认只回答：

- 今天有哪些股票发生了值得关注的变化
- 变化背后的数据和新闻证据是什么
- 用户后续应该观察哪些指标

产品默认不提供：

- 买入、卖出、持有等直接投资指令
- 收益承诺或确定性判断
- 自动下单或交易执行

## 3. 目标用户

### 3.1 个人投资者

有自选股列表，希望每天开盘前、盘中或收盘后快速了解重点股票变化。

### 3.2 研究型用户

关注某些行业、主题或公司，希望获得带证据链的摘要，而不是零散新闻标题。

### 3.3 团队用户

需要将股票异动、公告、财报和新闻摘要推送到企业微信、飞书、钉钉、Slack 或邮件群组。

## 4. 产品目标

### 4.1 MVP 目标

第一版聚焦三个高价值场景：

1. 自选股每日早报
2. 盘中异动预警
3. 收盘复盘摘要

### 4.2 成功指标

- 用户能在 10 分钟内完成自选股和推送渠道配置
- 每日早报能在预设时间稳定发送
- 每条推送都包含可回溯的数据或新闻来源
- 同一股票同一事件不会重复推送
- 推送内容不包含未经证据支持的价格、涨跌幅或新闻事实

## 5. 范围

### 5.1 MVP 范围

- 用户维护自选股列表
- 用户配置推送时间和推送渠道
- 系统定时采集行情、新闻、公告和基础财务数据
- 系统根据规则识别重要变化
- LLM 根据结构化证据生成摘要
- 合规与事实校验模块检查输出
- 系统记录推送历史，支持幂等去重

### 5.2 非 MVP 范围

- 自动交易
- 组合调仓建议
- 个性化收益预测
- 高频交易级实时行情
- 复杂多账户权限体系
- 移动端 App

## 6. 核心用户故事

### 6.1 每日早报

作为用户，我希望每天开盘前收到自选股早报，以便快速知道哪些股票有隔夜新闻、公告、财报或盘前异动。

验收标准：

- 用户可以配置早报发送时间
- 早报按用户自选股生成
- 早报包含重点股票、关键变化、证据来源和风险提示
- 无重要变化时，系统发送低噪音摘要，而不是强行制造结论

### 6.2 盘中异动预警

作为用户，我希望当自选股出现明显价格、成交量或新闻异动时收到预警，以便及时关注。

验收标准：

- 用户可以开启或关闭盘中预警
- 系统支持价格涨跌幅、成交量放大、突破高低点等规则
- 单个事件在冷却时间内不重复推送
- 推送必须包含触发规则和原始数据

### 6.3 收盘复盘

作为用户，我希望收盘后收到当天自选股复盘，以便了解当天主要变化和后续观察点。

验收标准：

- 系统在收盘后定时发送
- 摘要按重要性排序
- 每只重点股票包含行情变化、事件说明和后续观察指标
- 报告明确区分事实、推断和风险

## 7. 功能需求

### 7.1 用户配置

系统需要支持：

- 添加、删除、查看自选股
- 配置市场类型，例如 A 股、港股、美股
- 配置推送渠道
- 配置每日早报和收盘复盘时间
- 配置盘中预警开关
- 配置单股票或全局冷却时间

### 7.2 调度模块

系统需要支持：

- 定时任务触发
- 交易日识别
- 不同市场时区处理
- 任务入队
- 任务状态记录
- 失败重试

调度模块只负责创建任务，不直接生成报告或发送推送。

### 7.3 数据采集模块

系统需要采集：

- 实时或延迟行情
- 日线和分钟级行情摘要
- 成交量和均量
- 公司公告
- 财经新闻
- 财报和关键财务指标
- 财经日历事件

每条采集数据都需要记录来源、抓取时间和原始引用。

### 7.4 信号识别模块

系统需要通过确定性规则识别信号，避免让 LLM 直接猜测行情是否异常。

第一版规则包括：

- 日内涨跌幅超过阈值
- 成交量超过近 20 日均量阈值
- 突破近 20 日高点或低点
- 发布财报或重大公告
- 负面或正面新闻数量显著增加
- 盘前或盘后价格大幅波动

信号输出格式应包含：

- 股票代码
- 股票名称
- 触发规则
- 严重程度
- 证据数据
- 数据来源
- 触发时间

### 7.5 LLM 摘要模块

LLM 只接收结构化证据和允许引用的上下文，不直接访问未校验的外部数据。

摘要需要包含：

- 发生了什么
- 为什么值得关注
- 支撑证据
- 后续观察指标
- 风险提示

摘要需要避免：

- 无来源的事实
- 绝对化判断
- 投资指令
- 收益承诺

### 7.6 合规与事实校验模块

系统需要在发送前检查：

- 摘要中的价格、百分比和时间是否能回溯到证据
- 是否出现买入、卖出、稳赚、必涨等高风险表述
- 是否缺少风险提示
- 是否有重复事件
- 是否有空洞结论或证据不足的判断

校验失败时，系统应降级为只发送结构化事实摘要，或跳过该条推送并记录原因。

### 7.7 推送模块

第一版支持至少一种推送渠道，推荐优先支持企业微信、飞书或邮件。

推送内容格式：

- 标题
- 摘要
- 关键证据
- 风险提示
- 数据来源链接或引用
- 生成时间

### 7.8 推送历史

系统需要记录：

- 推送 ID
- 用户 ID
- 股票代码
- 触发事件
- 推送渠道
- 推送内容
- 发送状态
- 发送时间
- 去重键

## 8. 推荐系统架构

```text
Scheduler
  -> Task Queue
    -> Data Collector
      -> Signal Engine
        -> LLM Analyst
          -> Guardrail
            -> Push Service
              -> Delivery History
```

### 8.1 Scheduler

负责按时间表或事件条件生成任务，并写入任务队列。

### 8.2 Task Queue

负责削峰、重试和任务状态管理。MVP 可使用 Redis Queue、Celery、RQ 或轻量本地队列。

### 8.3 Data Collector

负责采集行情、公告、新闻和财务数据，并统一成内部结构。

### 8.4 Signal Engine

负责用规则和统计指标判断是否值得推送。

### 8.5 LLM Analyst

负责将结构化证据转换成用户可读摘要。

### 8.6 Guardrail

负责事实校验、合规校验、重复校验和降级策略。

### 8.7 Push Service

负责发送消息，并记录发送结果。

## 10. 非功能需求

### 10.1 可靠性

- 定时任务失败后支持重试
- 数据源不可用时应降级，而不是生成无依据摘要
- 推送失败需要记录错误原因

### 10.2 幂等性

- 同一用户、同一股票、同一事件在冷却时间内只推送一次
- 任务重试不能造成重复推送

### 10.3 可观测性

系统需要记录：

- 任务创建数量
- 任务成功率
- 数据源失败率
- LLM 生成失败率
- Guardrail 拦截数量
- 推送成功率

### 10.4 安全与隐私

- API Key 不应提交到仓库
- 用户配置和推送渠道凭证需要加密存储
- 日志中不记录完整敏感凭证

### 10.5 合规

- 默认文案必须包含风险提示
- 不提供确定性收益承诺
- 不提供未经授权的投资建议
- 数据源授权和使用条款需要单独确认

## 12. 风险与应对

### 12.1 数据质量风险

风险：行情或新闻数据延迟、缺失、错误。  
应对：记录数据来源和时间；低置信度时降级输出；关键字段缺失时跳过生成。

### 12.2 LLM 幻觉风险

风险：LLM 编造原因、数字或结论。  
应对：只允许基于结构化证据生成；发送前做事实回溯校验。

### 12.3 通知疲劳风险

风险：过多低价值消息导致用户关闭通知。  
应对：设置严重程度阈值、冷却时间和每日推送上限。

### 12.4 合规风险

风险：内容被理解为投资建议。  
应对：明确产品边界；限制高风险措辞；保留风险提示和证据来源。

## 13. 验收清单

- 用户可以配置至少 5 只自选股
- 系统可以按固定时间生成每日早报
- 系统可以识别至少 3 类异动信号
- 系统可以通过至少 1 个渠道发送推送
- 每条推送都有历史记录和去重键
- 摘要中的关键事实可以追溯到数据源
- Guardrail 能拦截明显投资建议式文案
- 数据源失败时系统不会生成无依据内容

## 14. 开放问题

- 第一版优先支持哪个市场：A 股、港股还是美股
- 第一版优先支持哪个推送渠道
- 行情和新闻数据源选择
- 是否需要多用户系统，还是先做单用户本地工具
- 是否需要 Web 配置界面，还是先用配置文件
