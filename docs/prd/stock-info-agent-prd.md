# 定时股票与指数信息推送 Agent PRD

版本：v0.1  
日期：2026-06-24  
状态：Draft  

## 1. 背景

股票和指数关注者通常需要在固定时间或关键事件发生时快速了解关注标的状态，但原始行情、公告、新闻和财务数据分散在不同平台。用户真正需要的不是更多信息流，而是经过筛选、可追溯、低噪音的市场信息摘要。

本产品旨在构建一个定时推送股票与指数信息的 Agent。它按用户配置的关注标的池、时间表和关注规则，自动采集行情与资讯，识别值得关注的变化，并生成结构化摘要推送到指定渠道。关注标的包括用户自选股和指数。

## 2. 产品定位

本产品是股票与指数信息监控和研究摘要工具，不是自动荐股、自动交易或投资顾问系统。

产品默认只回答：

- 今天有哪些关注标的发生了值得关注的变化
- 变化背后的数据和新闻证据是什么
- 用户后续应该观察哪些指标

产品默认不提供：

- 买入、卖出、持有等直接投资指令
- 收益承诺或确定性判断
- 自动下单或交易执行

## 3. 目标用户

### 3.1 个人投资者

第一版本面向单个个人投资者。有 A 股自选股和指数关注列表，希望每天开盘前、盘中或收盘后快速了解重点标的变化。

### 3.2 研究型用户

关注某些行业、主题、公司或市场指数，希望获得带证据链的摘要，而不是零散新闻标题。

### 3.3 团队用户

需要将股票和指数异动、公告、财报和新闻摘要推送到企业微信、飞书、钉钉、Slack 或邮件群组。团队协作不是第一版本重点，第一版本只要求单用户本地部署。

## 4. 产品目标

### 4.1 核心目标

产品聚焦三个高价值场景：

1. 关注标的每日早报
2. 盘中异动预警
3. 收盘复盘摘要

### 4.2 成功指标

- 用户能在 10 分钟内完成自选股、指数和推送渠道配置
- 每日早报能在预设时间稳定发送
- 重要信息能同时进入手机微信提醒和本地 Web 信息展示界面
- 每条推送都包含可回溯的数据或新闻来源
- 同一关注标的同一事件不会重复推送
- 推送内容不包含未经证据支持的价格、涨跌幅或新闻事实
- 盘中重要异动在数据源可用后 2 分钟内完成推送，P95 延迟可观测

## 5. 范围

### 5.1 支持范围

- 用户维护关注标的列表，包含自选股和指数
- 第一版本只支持 A 股股票和 A 股相关指数
- 第一版本采用单用户本地工具形态部署
- 用户配置推送时间和推送渠道
- 用户通过本地 Web 界面完成配置，并查看推送历史、异动事件和系统状态
- 系统定时采集股票和指数行情、新闻、公告和基础财务数据
- 系统根据规则识别重要变化
- LLM 根据结构化证据生成摘要
- 合规与事实校验模块检查输出
- 系统记录推送历史，支持幂等去重

### 5.2 边界外能力

- 自动交易
- 组合调仓建议
- 个性化收益预测
- 高频交易级实时行情
- 毫秒级交易信号或交易执行延迟保障
- 复杂多账户权限体系
- 多用户、团队权限、云端 SaaS 托管
- 港股、美股和其他海外市场
- 移动端 App

## 6. 核心用户故事

### 6.1 每日早报

作为用户，我希望每天开盘前收到关注标的早报，以便快速知道哪些自选股或指数有隔夜新闻、公告、财报、盘前异动或市场变化。

验收标准：

- 用户可以配置早报发送时间
- 早报按用户配置的自选股和指数生成
- 早报包含重点标的、关键变化、证据来源和风险提示
- 无重要变化时，系统发送低噪音摘要，而不是强行制造结论

### 6.2 盘中异动预警

作为用户，我希望当自选股或指数出现明显价格、成交量、成分股相关事件或新闻异动时收到预警，以便及时关注。

验收标准：

- 用户可以开启或关闭盘中预警
- 系统支持价格涨跌幅、成交量放大、突破高低点等规则
- 单个事件在冷却时间内不重复推送
- 推送必须包含触发规则和原始数据
- 重要异动从数据源可用到完成推送的 P95 延迟不超过 2 分钟

### 6.3 收盘复盘

作为用户，我希望收盘后收到当天关注标的复盘，以便了解自选股和指数的主要变化和后续观察点。

验收标准：

- 系统在收盘后定时发送
- 摘要按重要性排序
- 每个重点标的包含行情变化、事件说明和后续观察指标
- 报告明确区分事实、推断和风险

## 7. 功能需求

### 7.1 用户配置

系统需要支持：

- 添加、删除、查看自选股和指数
- 配置 A 股股票和 A 股相关指数标的
- 配置 Server酱 Turbo SendKey
- 配置企业微信群机器人 Webhook 作为可选备用通道
- 配置每日早报和收盘复盘时间
- 配置盘中预警开关
- 配置单个关注标的或全局冷却时间

### 7.2 调度模块

系统需要支持：

- 定时任务触发
- A 股交易日识别
- A 股交易时段处理
- 任务入队
- 任务状态记录
- 失败重试

调度模块只负责创建任务，不直接生成报告或发送推送。

### 7.3 数据采集模块

系统需要围绕用户配置的自选股和指数采集：

- 实时或延迟行情
- 日线和分钟级行情摘要
- 成交量和均量
- 公司公告
- 财经新闻
- 财报和关键财务指标
- 指数成分、权重和代表性成分股变化
- 财经日历事件

每条采集数据都需要记录来源、抓取时间和原始引用。

### 7.4 信号识别模块

系统需要通过确定性规则识别信号，避免让 LLM 直接猜测行情是否异常。

规则包括：

- 日内涨跌幅超过阈值
- 成交量超过近 20 日均量阈值
- 突破近 20 日高点或低点
- 发布财报或重大公告
- 负面或正面新闻数量显著增加
- 盘前或盘后价格大幅波动
- 指数大幅波动、突破关键区间或代表性成分股集中异动

信号输出格式应包含：

- 标的代码
- 标的名称
- 标的类型，例如股票或指数
- 触发规则
- 严重程度
- 证据数据
- 数据来源
- 触发时间

### 7.5 LLM 摘要模块

LLM 只接收结构化证据和允许引用的上下文，不直接访问未校验的外部数据。

摘要需要包含：

- 发生了什么
- 为什么值得关注
- 支撑证据
- 后续观察指标
- 风险提示

摘要需要避免：

- 无来源的事实
- 绝对化判断
- 投资指令
- 收益承诺

### 7.6 合规与事实校验模块

系统需要在发送前检查：

- 摘要中的价格、百分比和时间是否能回溯到证据
- 是否出现买入、卖出、稳赚、必涨等高风险表述
- 是否缺少风险提示
- 是否有重复事件
- 是否有空洞结论或证据不足的判断

校验失败时，系统应降级为只发送结构化事实摘要，或跳过该条推送并记录原因。

### 7.7 推送模块

系统推荐优先支持两种独立开发者可实现的推送方案。两种方案都应使用公开 HTTP API 或 Webhook，不使用个人微信逆向协议、外挂客户端或非官方登录态。

#### 7.7.1 方案一：Server酱 Turbo 微信推送

适用场景：

- 面向个人用户或小规模测试
- 独立开发者希望快速将消息推送到微信相关入口
- 需要低配置成本、低维护成本的提醒能力
- 作为第一版本默认主推送通道

实现方式：

- 用户在 Server酱 Turbo 获取 SendKey
- 系统通过 HTTPS API 提交标题、摘要和正文
- Server酱负责将消息转发到个人手机微信、企业微信或其他已绑定渠道

优点：

- 接入成本低，一个 HTTP 请求即可发送
- 不需要开发和审核微信小程序、公众号或 App
- 适合独立开发者、本地工具和早期产品验证

限制：

- 依赖第三方服务的配额、稳定性和可用性
- 适合提醒，不适合作为强 SLA 的唯一通道
- 用户需要完成 Server酱侧的绑定和授权

#### 7.7.2 方案二：企业微信群机器人 Webhook

适用场景：

- 面向团队、研究小组或个人创建的企业微信测试组织
- 希望消息进入企业微信群，便于多人查看和讨论
- 需要 Markdown 格式、群内留痕和较直接的 Webhook 集成

实现方式：

- 在企业微信群中添加群机器人并获取 Webhook URL
- 系统通过 HTTPS Webhook 推送文本或 Markdown 消息
- 推送内容包含标题、摘要、触发规则、关键证据和风险提示

优点：

- Webhook 集成简单，独立开发者可自行实现
- 适合团队协作和消息留痕
- 不依赖个人微信非官方接口

限制：

- 需要企业微信环境和群机器人配置
- 群消息可能受平台频率限制和群成员通知设置影响
- 不适合承载敏感凭证或完整长报告，长内容应只推送摘要和链接

推送内容格式：

- 标题
- 摘要
- 关键证据
- 风险提示
- 数据来源链接或引用
- 生成时间

推送失败时，系统应优先重试原渠道；连续失败后可降级到备用渠道，并在推送历史中记录失败原因和降级路径。

第一版本默认主通道为 Server酱 Turbo。企业微信群机器人 Webhook 作为可选备用通道，不要求用户必须配置。

### 7.8 推送历史

系统需要记录：

- 推送 ID
- 用户 ID
- 标的代码
- 标的类型
- 触发事件
- 推送渠道
- 推送内容
- 发送状态
- 发送时间
- 去重键
- 首次采集时间
- 信号触发时间
- 推送入队时间
- 推送完成时间
- 推送延迟

### 7.9 实时性保障

本产品的实时性目标是业务实时提醒，不承诺高频交易级实时行情或毫秒级延迟。实时性通过数据采集、信号识别、任务队列、推送通道和可观测性共同保障。

系统需要支持：

- 优先使用支持 WebSocket、实时订阅或增量更新的数据源
- 数据源不支持订阅时，在交易时段对重点关注标的进行短轮询
- 将盘中异动预警任务与日报、复盘任务拆分到不同优先级队列
- 异动任务跳过非必要的长链路分析，先生成结构化事实摘要，再补充完整分析
- Guardrail 使用轻量同步校验，避免 LLM 长生成阻塞重要预警
- 对推送通道设置超时、重试和备用通道
- 记录端到端延迟指标，包括采集、识别、入队、出队、发送完成时间

延迟目标：

- 重点标的行情轮询间隔：15-60 秒，具体取决于数据源限制和成本
- 信号识别与去重：10 秒内完成
- 摘要生成与校验：30 秒内完成；超时则降级为结构化事实摘要
- 推送发送：10 秒内提交到推送服务
- 盘中重要异动端到端 P95 延迟：数据源可用后 2 分钟内完成推送

实时性降级策略：

- 数据源延迟或不可用时，推送明确标注数据时间和延迟状态
- LLM 超时时，先推送结构化事实摘要，不等待完整分析
- 主推送通道失败时，自动切换到备用推送通道
- 同一事件在冷却时间内只保留最新证据，不重复刷屏

### 7.10 Web 配置与信息展示界面

第一版本需要提供本地 Web 界面，作为单用户配置入口和信息展示界面。Agent 可以部署在本机或本地服务器上，用户通过浏览器访问 Web 界面。

Web 界面需要支持：

- 配置 A 股自选股和指数关注列表
- 配置早报、盘中预警、收盘复盘时间
- 配置 Server酱 Turbo SendKey 和可选企业微信群机器人 Webhook
- 查看最近推送记录
- 查看盘中异动事件列表
- 查看每条摘要的证据来源、生成时间、推送状态和推送延迟
- 查看数据源、任务队列、LLM、Guardrail 和推送通道的最近运行状态

手机微信和 Web 界面的信息关系：

- 手机微信用于实时提醒，只发送高信号密度摘要和关键证据
- Web 界面用于完整查看，包括历史记录、详细证据、失败原因和系统状态
- 同一事件在手机微信和 Web 界面使用同一个事件 ID 和去重键，便于追溯

## 8. 推荐系统架构

```text
Scheduler
  -> Task Queue
    -> Data Collector
      -> Signal Engine
        -> LLM Analyst
          -> Guardrail
            -> Push Service
              -> Delivery History
              -> Web Dashboard
```

### 8.1 Scheduler

负责按时间表或事件条件生成任务，并写入任务队列。第一版本按 A 股交易日和交易时段运行。

### 8.2 Task Queue

负责削峰、优先级调度、重试和任务状态管理。可使用 Redis Queue、Celery、RQ 或轻量本地队列。盘中异动预警任务应使用高优先级队列，避免被日报、复盘或批量数据任务阻塞。

### 8.3 Data Collector

负责采集股票和指数行情、公告、新闻和财务数据，并统一成内部结构。

### 8.4 Signal Engine

负责用规则和统计指标判断是否值得推送。

### 8.5 LLM Analyst

负责将结构化证据转换成用户可读摘要。

### 8.6 Guardrail

负责事实校验、合规校验、重复校验和降级策略。

### 8.7 Push Service

负责发送消息，并记录发送结果。第一版本默认通过 Server酱 Turbo 推送到手机微信，企业微信群机器人 Webhook 作为可选备用通道。

### 8.8 Web Dashboard

负责提供本地 Web 配置和信息展示能力。Web Dashboard 读取用户配置、推送历史、信号事件、证据来源和运行状态，不直接生成信号或摘要。

### 8.9 Storage

第一版本默认使用本地 SQLite 存储，降低单用户本地部署复杂度。存储层需要预留后续切换到 MySQL 的能力，业务逻辑不得直接依赖 SQLite 专有 SQL 或文件路径。

存储设计要求：

- 通过 Repository 或 Store 接口访问数据，不在业务模块中直接拼接 SQL
- 使用 ORM 迁移工具管理表结构版本
- 数据表字段类型优先选择 SQLite 和 MySQL 都能表达的通用类型
- 所有 JSON 字段需要定义序列化边界，避免依赖某一数据库的 JSON 专有查询能力
- 数据库连接通过配置项管理，后续可从 `sqlite:///local.db` 切换为 MySQL 连接串

## 10. 非功能需求

### 10.1 可靠性

- 定时任务失败后支持重试
- 数据源不可用时应降级，而不是生成无依据摘要
- 推送失败需要记录错误原因

### 10.2 幂等性

- 同一用户、同一关注标的、同一事件在冷却时间内只推送一次
- 任务重试不能造成重复推送

### 10.3 可观测性

系统需要记录：

- 任务创建数量
- 任务成功率
- 数据源失败率
- LLM 生成失败率
- Guardrail 拦截数量
- 推送成功率
- 推送端到端 P50、P95、P99 延迟
- 各推送渠道失败率和降级次数

### 10.4 安全与隐私

- API Key 不应提交到仓库
- 用户配置和推送渠道凭证需要加密存储
- 日志中不记录完整敏感凭证

### 10.5 数据可迁移性

- 第一版本默认使用 SQLite 本地数据库
- 数据访问必须经过 Repository 或 Store 抽象
- 表结构迁移脚本需要兼容后续 MySQL 部署
- 不在核心业务逻辑中使用 SQLite 专有语法
- 后续切换 MySQL 时，应通过配置和迁移完成，不重写核心业务模块

### 10.6 合规

- 默认文案必须包含风险提示
- 不提供确定性收益承诺
- 不提供未经授权的投资建议
- 数据源授权和使用条款需要单独确认

## 11. 实施里程碑

### Milestone 1：基础配置与每日早报

## 12. 风险与应对

### 12.1 数据质量风险

风险：行情或新闻数据延迟、缺失、错误。  
应对：记录数据来源和时间；低置信度时降级输出；关键字段缺失时跳过生成。

### 12.2 LLM 幻觉风险

风险：LLM 编造原因、数字或结论。  
应对：只允许基于结构化证据生成；发送前做事实回溯校验。

### 12.3 通知疲劳风险

风险：过多低价值消息导致用户关闭通知。  
应对：设置严重程度阈值、冷却时间和每日推送上限。

### 12.4 实时性不稳定风险

风险：数据源延迟、LLM 生成耗时、队列拥塞或推送渠道限流导致重要异动延迟。  
应对：拆分高优先级预警队列；设置 LLM 超时和结构化摘要降级；记录端到端延迟；主通道失败时切换备用通道。

### 12.5 合规风险

风险：内容被理解为投资建议。  
应对：明确产品边界；限制高风险措辞；保留风险提示和证据来源。

## 13. 验收清单

- 用户可以配置至少 5 个关注标的，包含自选股或指数
- 用户可以通过本地 Web 界面完成关注标的、推送通道和提醒时间配置
- 系统可以按固定时间生成每日早报
- 系统可以识别至少 3 类异动信号
- 系统默认通过 Server酱 Turbo 将消息推送到手机微信
- Web 界面可以展示推送历史、异动事件、证据来源和系统运行状态
- 每条推送都有历史记录和去重键
- 摘要中的关键事实可以追溯到数据源
- Guardrail 能拦截明显投资建议式文案
- 数据源失败时系统不会生成无依据内容
- 盘中重要异动端到端 P95 推送延迟不超过 2 分钟，且可从日志中追溯
